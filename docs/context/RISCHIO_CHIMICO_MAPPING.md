## 3. The model вЂ” MoVaRisCh

Algorithm proposed by Regioni **Toscana, Emilia-Romagna, Lombardia**, satisfying **art. 223 c.1 D.Lgs. 81/08** (Titolo IX Capo I). It classifies each exposure above/below the *"soglia del rischio irrilevante per la salute"*.

```
Rischio salute:    R = P Г— E
  Rinal = P Г— Einal            (inhalation route)
  Rcute = P Г— Ecute            (dermal route)
  Rcum  = в€љ(RinalВІ + RcuteВІ)   (when BOTH routes apply)
Ranges: 0.1 в‰¤ Rinal в‰¤ 100 В· 1 в‰¤ Rcute в‰¤ 100 В· 1 в‰¤ Rcum в‰¤ 141
```

### 3.1 P вЂ” Indice di PericolositГ  (hazard)
- P = **highest score** among the substance's **H-phrases** (Reg. CLP 1272/2008) вЂ” or legacy **R-phrases** (D.Lgs. 52/97 & 65/03) when CLP not available.
- Full score tables extracted to the seed: **103 H-codes + 19 mixture rules**, **125 R-codes + 19 mixture rules**.
- **Cancerogeni / mutageni** (H340/H350/H350i/H351, R45/R46/R49) are flagged `CANCEROGENO` and **excluded** from P вЂ” they fall under Titolo IX **Capo II**, where no irrelevance threshold exists. (Surface a warning; do not score.)
- Multi-phrase example: EMAX RESINA EP 138 в†’ H302(2.0), H315(2.5), H317 cat.1B(4.5), H319(3.0), H411(1.0), EUH205(4.5) в‡’ **P = 4.5**. вњ… matches the document.

### 3.2 Einal = I Г— d (inhalation)
`I` (sub-indice di intensitГ ) is derived through **4 cascading matrices**; `d` scales by worker distance from source.

| Step | Inputs | в†’ Indicator | Values |
|------|--------|-------------|--------|
| Matrix 1 | proprietГ  chimico-fisiche Г— quantitГ  in uso | **D** (disponibilitГ ) | Bassa=1 В· Medio/Bassa=2 В· Medio/Alta=3 В· Alta=4 |
| Matrix 2 | D Г— tipologia d'uso | **U** | Basso=1 В· Medio=2 В· Alto=3 |
| Matrix 3 | U Г— tipologia di controllo | **C** (compensazione) | Basso=1 В· Medio=2 В· Alto=3 |
| Matrix 4 | C Г— tempo di esposizione | **I** (intensitГ ) | Bassa=1 В· Medio/Bassa=3 В· Medio/Alta=7 В· Alta=10 |
| `d` | distanza dalla sorgente | **d** | <1m=1 В· 1вЂ“3m=0.75 В· 3вЂ“5m=0.5 В· 5вЂ“10m=0.25 В· в‰Ґ10m=0.1 |

A modified `Matrice 1bis/2bis` variant exists for agents *generated by* a work process (welding, hot plastics). Out of scope for v1 unless a client needs it; noted in the seed.

### 3.3 Ecute (dermal)
Single matrix: **tipologia d'uso Г— livello di contatto cutaneo** в†’ Basso=1 В· Medio=3 В· Alto=7 В· Molto Alto=10.

Dermal assessment is **mandatory** when any of: an H/R phrase expressly indicates a dermal hazard/absorption; the SDS notes skin absorption; the substance has an OEL with a "pelle/skin" notation; or direct contact is possible in the activity. Otherwise inhalation-only.

### 3.4 Health classification (zones)
| R (max of routes) | Zona | Classificazione |
|-------------------|------|-----------------|
| 0.1 в‰¤ R < 15 | **VERDE** | Rischio irrilevante per la salute |
| 15 < R < 21 | **ARANCIO** | Irrilevante вЂ” zona di incertezza (rivedere punteggi + medico competente) |
| 21 в‰¤ R в‰¤ 40 | **GIALLO** | Superiore all'irrilevante (artt. 225,226,229,230) |
| 40 < R в‰¤ 80 | **ROSSA** | Superiore вЂ” rischio elevato |
| R > 80 | **NERA** | Superiore вЂ” grave rischio |

### 3.5 Safety level (rischio per la sicurezza)
Independent of the health score: each H/R code maps to **`Basso`** or **`Non Basso`** (229-code table extracted). The substance's level = **worst** across its phrases (any `Non Basso` в‡’ `Non Basso`). Example: EMAX has H319 в‡’ `Non Basso`. вњ… matches.

## 4. вљ пёЏ The CORRECT calculation rule (decision: "make correct rule")

Luca's source documents were produced by a third-party tool that contains a **data-entry artifact we must NOT replicate**. Evidence вЂ” across all 30 schede in both files:

- Whenever physical state was left as generic **`"Liquido"`** (not one of the 4 documented volatility tiers), the tool emitted **`D = 0` в‡’ `Einal = 0` в‡’ `Rinal = 0`**, silently dropping the inhalation route. A few schede also show **`d = 0`** from a blank distance field.
- Substances correctly classified as `Solido`, `Stato gassoso`, or `Polveri fini` computed normally (Dв‰Ґ1, real Einal).

Both `D=0` and `d=0` **contradict the model's own stated ranges** (`0.1 в‰¤ Rinal в‰¤ 100`; `d в€€ {1,0.75,0.5,0.25,0.1}`). They are missing-input defaults, not a real "no inhalation" rule вЂ” liquids always present an inhalation pathway proportional to volatility.

**Correct rule (what we implement):**

1. **Physical state must always resolve to one of the 4 availability tiers.** A liquid is never just `"Liquido"` вЂ” classify it as `Bassa VolatilitГ ` or `Media/Alta VolatilitГ ` from the SDS (vapour pressure / boiling point). в†’ This is the primary job of the AI-suggester (В§6).
2. **`Einal = I Г— d` is always > 0** (minimum `1 Г— 0.1 = 0.1`). Never zero out the inhalation route.
3. **Distance always maps to a real `d`.** Default to the conservative worst case `d = 1` (<1 m) when unknown вЂ” never `0`.
4. Everything else follows the documented matrices exactly (В§3).

**Impact check (faithful to the spirit, safe vs. their output):** re-running the correct rule on Luca's schede keeps every result in the **same final zone** (these are all `contenimento completo`, `<15 min`, so `I=1` в†’ small Rinal that doesn't cross 15), e.g. EMAX: their `Rcum=13.5` в†’ correct `Rcum=в€љ(4.5ВІ+13.5ВІ)=14.23`, still **Irrilevante**. The correction is invisible for low-exposure cases but **prevents dangerous under-reporting** for higher-exposure scenarios (longer time / direct manipulation) where their `D=0` bug would wrongly zero a real inhalation risk. Worth flagging to Luca that our intermediate `Rinal` will be non-zero where his old tool printed 0.

## 5. Data model вЂ” assessment per **worker Г— substance** (decision)

Each exposed worker is assessed against **each substance** they use (a per-`(persona, sostanza)` scheda), consistent with the existing per-worker philosophy (cf. `Persona.dpi_codes` / `rischi_specifici_codes`).

New table **`rischio_chimico_esposizioni`** (one row = one scheda):

| Field | Source |
|-------|--------|
| `id`, `azienda_id` | вЂ” |
| `persona_id` в†’ `persone.id` | worker (nominativo, mansione) |
| `sostanza_id` в†’ `sostanze_chimiche.id` | substance (frasi_h, produttore, stato) |
| **Inputs (AI-suggested, human-reviewed):** `proprieta_fisiche` (tier), `quantita_classe`, `tipologia_uso`, `tipologia_controllo`, `tempo_esposizione`, `distanza_classe`, `contatto_cutaneo`, `via_cutanea_applicabile` (bool) | В§6 |
| **Computed (stored for traceability):** `p_score`, `d_ind`, `u_ind`, `c_ind`, `i_ind`, `einal`, `rinal`, `ecute`, `rcute`, `rcum`, `livello_salute` (zona), `livello_sicurezza` | calculator |
| `ai_suggested` (bool), `human_reviewed` (bool), `created_at` | lifecycle |

Reuse `Persona` (nominativo, mansione, codice_fiscale), `persone_ambienti` (M2M), and `SostanzaChimica` (already stores `frasi_h`, `produttore`, `stato_miscela`, `attivita_uso`) as-is. The **results table** in the doc groups by worker (`nominativo` + `mansione` + substances + livello salute/sicurezza); the per-substance **scheda** is the granular 20-row block.

> **Privacy:** `codice_fiscale` appears in the printed doc but must **never** reach the AI. The suggester needs only SDS-derived fields + ambiente type + `mansione` (a job title, not PII).

## 6. AI suggestion (decision: "ai suggested")

The 7 exposure inputs are **AI-suggested then human-reviewed** (the "review-only" principle). Use the existing `services/ai/client.py` Responses-API helpers + `OPENAI_MODEL_MEASURES` (`gpt-5.4-mini`, domain reasoning), structured output via Pydantic `text_format`.

- **Input to model (PII-stripped):** substance `nome_prodotto`, `stato_miscela`, `frasi_h`, `attivita_uso`; the worker's `mansione`; the `ambiente.tipo`; (optionally vapour pressure if captured from SDS).
- **Output (enums only):** the 7 fields in В§5, each constrained to its documented option set, with a short Italian rationale + confidence. **Default conservative** on uncertainty (worst-case tier / `d=1`), per the model's own guidance ("nell'incertezza вЂ¦ le condizioni che portano alla situazione peggiore").
- The **volatility-tier inference for liquids** (В§4 rule 1) is the highest-value suggestion вЂ” it's exactly the field the legacy tool fumbled.
- Output flows into the survey UI as pre-filled, editable values (mirror `SectorSuggestions` / SDS-extraction review UX). Nothing is final without operator confirmation.
